PHASE 486 — STEP 10
INSERTION BOUNDARY DEFINITION

────────────────────────────────

OBJECTIVE

Define the safest exact insertion boundary inside:

app/demo-runtime/page.tsx

for passive governance trace rendering.

This is a definition step only.
No UI mutation occurs here.

────────────────────────────────

INSPECTION BASIS

Source:
docs/phase486_step9_selected_ui_host_inspection.txt

Observed structure:

• Existing page component already renders the full demo surface
• Existing runtime result section begins at the report-gated render block
• Existing summary badges and explanation surfaces already exist
• Current file already contains local UI state and fetch logic
• New mutation must avoid changing existing control flow

────────────────────────────────

SAFE INSERTION BOUNDARY

Primary insertion zone:

INSIDE the existing:

{report ? ( ... ) : null}

runtime result section

AND

AFTER the summary badge block

AND

BEFORE downstream detailed result sections

────────────────────────────────

WHY THIS IS THE CORRECT BOUNDARY

• Governance trace is part of result consumption
• It should appear only when report exists
• It does not need new fetch behavior
• It avoids changing request-entry controls
• It avoids changing runDemo control flow
• It stays inside the operator-visible result surface

────────────────────────────────

BOUNDARY RULE

Allowed mutation shape:

• Add one passive render subsection
• Read only from existing report object
• Do not alter runDemo
• Do not alter prompt controls
• Do not alter approval controls
• Do not alter fetch behavior

────────────────────────────────

PROHIBITED INSERTION ZONES

Do NOT insert governance trace render:

• Above request input controls
• Inside runDemo()
• Inside useMemo governanceExplanation logic
• Inside checkbox controls
• Inside fetch request body
• Outside the existing report-gated result surface

────────────────────────────────

RENDER DEPENDENCY RULE

Governance trace panel may render ONLY from already-available page data.

If exact trace fields are absent from report:

• Do not fabricate trace data
• Do not derive simulated rule lists
• Do not invent evaluation items

In that case:

• render explicit "NO DATA"

────────────────────────────────

TARGET IMPLEMENTATION STYLE

Preferred shape:

Runtime Result
  ├─ Summary badges
  ├─ Governance Trace panel
  └─ Existing remaining result sections

This preserves structural order:

Admission / governance visibility first
Execution / reporting visibility after

────────────────────────────────

COMPLETION CONDITION

Insertion boundary is complete when:

• Exact render host is known
• Exact location inside the host is known
• Mutation can proceed as a single UI-only change
• No backend dependency changes are required

────────────────────────────────

CLASSIFICATION

INSERTION BOUNDARY DEFINED
