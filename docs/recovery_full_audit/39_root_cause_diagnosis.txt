PHASE 457 - ROOT CAUSE DIAGNOSIS
================================

WHY THE DASHBOARD KEEPS LOOKING OLDER
-------------------------------------

This did NOT happen because the recovery work was missing.

It happened because the recovery process proved two different things, and they were easy to confuse:

1. We proved that later commits and branches were recoverable.
2. We later proved that the specific recovered candidates were still serving the same older dashboard shell.

What the evidence now shows:

A. THE FIRST RECOVERY TARGETS WERE REAL, BUT THEY WERE THE WRONG VISUAL TARGETS
- phase65_layout and phase65_wiring were successfully recovered.
- operator_guidance was also recovered.
- But served-artifact verification later proved:
  - phase65_layout live HTML == phase65_wiring live HTML
  - phase65_layout live bundle.js == phase65_wiring live bundle.js
  - operator_guidance changed bundle.js but still served the same live HTML shell
- That means the remembered visual state was NOT actually in those checkpoints.

So the scans were not "wrong" in finding recoverable work.
They were wrong only in assuming those checkpoints corresponded to the remembered dashboard shell.

B. WE WERE RESTORING WIRING / FEATURE CORRIDORS, NOT THE TRUE VISUAL SHELL DIVERGENCE
The early recovery scans were heavily biased toward:
- telemetry wiring
- operator guidance wiring
- later cognition / FL-3 adjacency
- dashboard-related commits in general

But later proof showed the actual visible dashboard shell remained unchanged across those candidates.

That means we were restoring:
- valid later functionality corridors
without yet proving:
- actual shell/layout divergence

In other words:
we recovered real work,
but not the work that changed the visible shell the way you remembered.

C. THE RUNTIME FAILURES MASKED THE REAL ANSWER FOR TOO LONG
Several launch issues delayed clear visual proof:
- postgres host port 5432 collisions
- inherited dashboard host port 8080 collisions
- missing runtime seed tables
- malformed temporary compose overrides
- created containers with no working listener

Because of that, for a while we were spending effort proving whether candidates could boot,
before we had final proof of whether they actually served different shell artifacts.

Once the candidates finally booted, the artifact-hash proof became clear:
the served HTML shell for the selected candidates was still the same old shell.

D. THE REAL VISUAL TARGET IS OLDER / DIFFERENT THAN THE FIRST CHOSEN RECOVERY BAND
Later shell-divergence scanning found older shell candidates with genuinely different HTML hashes and marker sets.

That is the key correction.

The problem is no longer:
"Did the work survive?"

The problem is:
"Which commit actually introduced the shell you remember?"

Current best interpretation:
- the first recovery corridor restored valid later work
- that later work mostly affected JS behavior, telemetry, and guidance surfaces
- the remembered layout lives at an earlier shell-divergence checkpoint, not in the first recovery band we tested

CONCLUSION
----------

Why this is happening:

Because the initial recovery effort successfully restored recoverable commits,
but those commits were not the commits that changed the visible dashboard shell you are trying to get back.

Stated plainly:

- We recovered the wrong class of truth first.
- We proved recoverability before proving visual identity.
- The scans found real branches and real work.
- But the served-artifact proof later showed those branches still rendered the older shell.

So this is not a disappearance problem anymore.
It is a checkpoint identification problem.

WHAT IS NOW KNOWN WITH HIGH CONFIDENCE
--------------------------------------

1. Your work is not gone.
2. The first recovered candidates are real.
3. Those candidates are not the remembered visual target.
4. The remembered target must be selected from the true shell-divergence commits,
   not from the first layout/wiring corridor we tested.
5. The next step should stay tightly focused on shell-divergence candidates only.

NEXT SAFE DECISION RULE
-----------------------

Do not use broad dashboard-related history as the selection basis anymore.

Only trust checkpoints that prove one of these:
- different served HTML shell
- different visible marker set
- direct visual match to the remembered dashboard

Anything else may be recoverable,
but it is not necessarily the correct visual restore target.
